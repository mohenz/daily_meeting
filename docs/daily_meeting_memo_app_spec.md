# 데일리 미팅 메모 앱 기능명세

## 1. 문서 목적

- 본 문서는 데일리 미팅 메모 앱의 기능 범위와 운영 규칙을 정의한다.
- 본 문서는 디자인, UI 스타일, 색상, 컴포넌트 룩앤필을 다루지 않는다.
- 본 문서는 MVP 기준 기능 요구사항과 권한 정책, 데이터 처리 규칙을 정리한다.
- 본 문서는 현재 단계의 단순 사용자 관리 방안과 향후 플랫폼 확장 시 보안 고도화 방향을 함께 정리한다.

## 2. 시스템 목표

- 날짜별, 작업자별로 데일리 미팅 내용을 등록하고 조회할 수 있어야 한다.
- 하루 기본 미팅 2종 `Daily Scrum`, `Daily Wrap-Up`을 운영할 수 있어야 한다.
- 필요 시 같은 날짜에 `Emergency` 미팅을 추가할 수 있어야 한다.
- 작업자는 본인 작업현황을 직접 등록하고 수정할 수 있어야 한다.
- 관리자는 작업자 관리, 미팅 생성, 전체 현황 관리 기능을 수행할 수 있어야 한다.

## 3. 사용자 역할

### 3.1 작업자

- 본인 작업자 프로필로 앱을 사용한다.
- 본인 미팅 메모를 등록할 수 있다.
- 본인 미팅 메모를 수정할 수 있다.
- 다른 작업자 현황을 조회할 수 있다.
- 작업자 정보, 미팅 세션, 타인 메모 삭제는 수행할 수 없다.

### 3.2 관리자

- 작업자의 모든 기능을 포함한다.
- 작업자를 등록할 수 있다.
- 작업자 상태를 변경할 수 있다.
- 작업자 정렬순위를 변경할 수 있다.
- 관리자 여부를 지정할 수 있다.
- 기본 미팅 생성과 긴급 미팅 추가를 수행할 수 있다.
- 전체 미팅 메모를 등록, 수정, 삭제할 수 있다.

## 4. 핵심 업무 개념

### 4.1 미팅 종류

- `Daily Scrum`
  - 하루 시작 시점의 진행 현황 공유 미팅
- `Daily Wrap-Up`
  - 하루 종료 시점의 마감 현황 공유 미팅
- `Emergency`
  - 긴급 상황 발생 시 추가 생성하는 임시 미팅

### 4.2 작업현황 항목

- 진행완료작업
- 진행중인작업
- 진행예정인작업
- 이슈사항

## 5. 기능 범위

## 5.1 계정 생성 및 로그인

### FR-01 회원가입

- 사용자는 `이름`, `이메일`, `비밀번호` 입력으로 계정을 생성할 수 있어야 한다.
- 별도의 OTP, 매직링크, 이메일 인증 발송 절차는 사용하지 않는다.
- 동일 이메일의 작업자가 이미 존재하고 비밀번호가 아직 설정되지 않았다면, 회원가입 시 초기 비밀번호를 설정하며 해당 작업자 계정과 연결한다.
- 동일 이메일의 작업자가 이미 존재하고 비밀번호까지 설정되어 있다면 중복 가입을 허용하지 않는다.
- 신규 이메일이면 새 작업자 프로필을 생성한다.

### FR-02 로그인

- 사용자는 `이메일`, `비밀번호`로 로그인할 수 있어야 한다.
- 비밀번호는 평문이 아니라 bcrypt 해시 기준으로 저장하고 비교한다.
- 비활성 계정은 로그인할 수 없어야 한다.

### FR-03 세션 유지

- 브라우저 재접속 시 기존 세션이 유지되면 다시 가입하지 않고 앱을 사용할 수 있어야 한다.
- 로그아웃 또는 세션 초기화 후에는 다시 이메일과 비밀번호로 로그인할 수 있어야 한다.

## 5.2 날짜 목록 관리

### FR-04 날짜 목록 조회

- 시스템은 등록된 미팅 날짜를 날짜 기준 내림차순으로 출력해야 한다.
- 날짜별로 다음 정보를 요약 표시해야 한다.
  - 등록 날짜
  - `Daily Scrum` 등록 인원 수
  - `Daily Wrap-Up` 등록 인원 수
  - `Emergency` 미팅 수
  - 최종 수정 시각

### FR-05 날짜 선택

- 사용자가 날짜를 선택하면 해당 날짜 기준의 작업자 리스트와 상세 현황을 조회할 수 있어야 한다.

## 5.3 미팅 세션 관리

### FR-06 기본 미팅 생성

- 관리자는 특정 날짜에 대해 `Daily Scrum`, `Daily Wrap-Up` 기본 세션 2개를 생성할 수 있어야 한다.
- 같은 날짜에는 `Daily Scrum` 1건, `Daily Wrap-Up` 1건만 허용한다.

### FR-07 긴급 미팅 생성

- 관리자는 특정 날짜에 `Emergency` 미팅을 여러 건 생성할 수 있어야 한다.
- 같은 날짜의 `Emergency`는 순번 또는 식별값으로 구분되어야 한다.

## 5.4 작업자 목록 조회

### FR-08 작업자 리스트 조회

- 사용자는 선택한 날짜의 작업자 리스트를 조회할 수 있어야 한다.
- 작업자 리스트는 `sort_order ASC`, `name ASC` 기준으로 정렬해야 한다.
- 작업자별로 다음 상태를 표시해야 한다.
  - `Daily Scrum` 등록 여부
  - `Daily Wrap-Up` 등록 여부
  - `Emergency` 등록 건수

## 5.5 작업현황 등록 및 수정

### FR-09 본인 작업현황 등록

- 작업자는 본인에게 해당하는 미팅 메모를 등록할 수 있어야 한다.
- 등록 대상 미팅은 선택 날짜의 각 미팅 세션이다.
- 입력 항목은 다음 4개다.
  - 진행완료작업
  - 진행중인작업
  - 진행예정인작업
  - 이슈사항

### FR-10 본인 작업현황 수정

- 작업자는 본인이 작성한 미팅 메모를 수정할 수 있어야 한다.
- 수정 시 기존 값이 화면에 미리 채워져야 한다.

### FR-11 관리자 대행 등록 및 수정

- 관리자는 모든 작업자의 미팅 메모를 등록할 수 있어야 한다.
- 관리자는 모든 작업자의 미팅 메모를 수정할 수 있어야 한다.

### FR-12 관리자 삭제

- 관리자는 미팅 메모를 삭제할 수 있어야 한다.
- 삭제는 이력 보존을 위해 soft delete 방식으로 처리한다.
- 작업자는 삭제 기능을 사용할 수 없다.

## 5.6 작업자 관리

### FR-13 작업자 등록

- 관리자는 작업자를 등록할 수 있어야 한다.
- 등록 항목은 다음과 같다.
  - 이름
  - 이메일
  - 초기 비밀번호(선택)
  - 역할명
  - 관리자 여부
  - 상태
  - 정렬순위

### FR-14 작업자 수정

- 관리자는 작업자 정보를 수정할 수 있어야 한다.
- 수정 가능 항목은 이름, 이메일, 역할명, 관리자 여부, 상태, 정렬순위, 비밀번호 재설정이다.

### FR-15 작업자 상태 관리

- 작업자 삭제 대신 `active`, `inactive` 상태 변경으로 운영한다.
- `inactive` 작업자는 기본 작업자 리스트에서 제외한다.

## 5.7 조회 및 상세 확인

### FR-16 작업자 상세 현황 조회

- 사용자는 특정 날짜와 특정 작업자를 선택해 미팅별 작업현황을 확인할 수 있어야 한다.
- 같은 날짜에 `Emergency`가 여러 건 있으면 각각 별도 카드 또는 목록으로 구분되어야 한다.

### FR-17 공통 조회 권한

- 모든 사용자는 활성 작업자 목록과 날짜별 현황을 조회할 수 있어야 한다.
- 단, 등록/수정/삭제 권한은 역할에 따라 제한한다.

## 7. 데이터 규칙

### DR-01 작업자 식별

- 작업자는 이메일 기준으로 식별한다.
- 이메일은 공백 제거 및 소문자 기준으로 정규화해 저장한다.

### DR-02 비밀번호 저장 방식

- 비밀번호는 bcrypt 해시로 저장한다.
- 관리자 사전 등록 작업자는 초기 단계에서 비밀번호가 비어 있을 수 있다.
- 비밀번호가 비어 있는 작업자는 회원가입 시 초기 비밀번호를 설정한다.

### DR-03 미팅 메모 유일성

- 한 작업자는 같은 미팅 세션에 대해 활성 메모 1건만 가질 수 있다.

### DR-04 기본 미팅 유일성

- 한 날짜에는 `Daily Scrum`, `Daily Wrap-Up` 각 1건만 존재할 수 있다.

### DR-05 Emergency 다건 허용

- 한 날짜에는 `Emergency` 미팅을 여러 건 둘 수 있다.

### DR-06 삭제 정책

- 작업자 삭제는 지원하지 않는다.
- 미팅 메모 삭제는 관리자만 가능하며 soft delete로 처리한다.

## 9. 운영 제약사항

- 앱은 내부 운영용 단순 입력 도구를 전제로 한다.
- 이메일 인증은 수행하지 않는다.
- 현재 단계의 사용자 관리 방식은 `defect_manage`와 유사한 앱 내부 사용자 관리 방식으로 단순화하는 것을 기준으로 한다.
- Supabase Auth + RLS 기반 사용자 보안은 스크럼보드가 플랫폼 구조로 확대될 때 2단계로 도입한다.
- 현재 단계는 데이터베이스 레벨 RLS를 적용하지 않고 앱 서비스 계층에서 권한을 통제한다.
- 최초 관리자 1명은 앱이 아니라 `daily_meeting_admin_seed.sql` 또는 SQL 직접 등록으로 먼저 생성한다.

## 9.1 보안 단계 계획

### Phase 1. 단기 운영 단계

- 내부 사용자 중심의 단순 운영을 우선한다.
- 사용자 관리 방식은 `defect_manage`와 유사한 앱 내부 계정/권한 관리 구조를 기준으로 한다.
- 관리자와 작업자 권한은 앱 서비스 계층에서 제어한다.
- 이 단계에서는 복잡한 외부 인증, OTP, 매직링크, 다중 테넌트 격리를 우선 범위에 포함하지 않는다.

### Phase 2. 플랫폼 확장 단계

- 여러 프로젝트와 조직 단위 운영이 필요한 시점에 Supabase Auth를 도입한다.
- 프로젝트 단위 데이터 격리와 역할 기반 접근 통제를 위해 RLS를 적용한다.
- `auth.users`와 앱 사용자 테이블의 연결, 프로젝트 멤버십, 감사 로그, 관리자 범위 통제를 함께 재설계한다.
- 이 단계에서 프로젝트별 데이터 노출 방지, 조직별 접근 분리, 관리자 범위 제한을 정식 보안 요구사항으로 승격한다.

## 10. MVP 범위

- 이메일 + 비밀번호 로그인
- 이름 + 이메일 + 비밀번호 회원가입
- 날짜 목록 조회
- 날짜 선택 후 작업자 리스트 조회
- 기본 미팅 2종 생성
- 긴급 미팅 추가
- 본인 미팅 메모 등록/수정
- 관리자용 전체 메모 등록/수정/삭제
- 관리자용 작업자 등록/수정/상태 관리

## 11. 디자인 제외 범위

- 컬러 시스템
- 타이포그래피
- 컴포넌트 스타일
- 카드/테이블 레이아웃 스타일
- 반응형 UI 세부 설계

위 항목은 별도 디자인 작업에서 정의한다.
