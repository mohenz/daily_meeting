# 공통 스크럼보드 플랫폼 설계안

## 1. 목적
본 문서는 특정 프로젝트에 종속되지 않는 공통 스크럼보드 플랫폼의 화면 구조, 운영 흐름, 데이터 모델, 구현 우선순위를 정의합니다. 이 플랫폼은 여러 프로젝트의 정보를 등록하고 프로젝트별 리소스를 연결한 뒤, 각 프로젝트 단위로 스크럼 미팅과 업무 추적을 운영할 수 있는 협업 도구를 목표로 합니다.

## 2. 플랫폼 설계 원칙
- 특정 프로젝트의 업무 유형에 종속되지 않아야 합니다.
- 여러 프로젝트를 한 플랫폼 안에서 독립적으로 운영할 수 있어야 합니다.
- 프로젝트별 목표, 일정, 태스크, 이슈를 분리 관리해야 합니다.
- 프로젝트별 문서, 저장소, 이슈트래커, 배포환경 등 리소스를 등록하고 빠르게 접근할 수 있어야 합니다.
- 스크럼 미팅 진행용 화면과 미팅 이후 실무 관리용 화면이 같은 데이터를 사용해야 합니다.
- 템플릿 기반으로 신규 프로젝트를 빠르게 개설할 수 있어야 합니다.

## 3. 핵심 개념

### 3.1 플랫폼 단위
플랫폼은 여러 프로젝트를 담는 상위 작업공간입니다.

주요 역할:
- 프로젝트 목록 관리
- 공통 사용자/권한 관리
- 공통 보드 템플릿 관리
- 공통 상태값, 라벨, 이슈 유형 관리

### 3.2 프로젝트 단위
프로젝트는 스크럼 미팅의 실제 운영 단위입니다.

프로젝트별 관리 대상:
- 프로젝트 기본 정보
- 프로젝트 팀원
- 프로젝트 리소스
- 활성 스프린트 또는 운영 주기
- 프로젝트 목표
- 프로젝트 태스크
- 프로젝트 이슈와 리스크

### 3.3 프로젝트 리소스
프로젝트 리소스는 해당 프로젝트 수행에 필요한 외부/내부 참조 정보를 의미합니다.

예시:
- 형상관리 저장소
- 요구사항 문서
- 화면설계서
- API 명세
- 결함관리 시스템
- 이슈 트래커
- 배포 URL
- 테스트 환경 URL
- 모니터링 대시보드
- 회의 문서

## 4. 사용자와 권한

### 4.1 사용자 유형
| 사용자 | 역할 |
| --- | --- |
| 플랫폼 관리자 | 플랫폼 설정, 템플릿 관리, 프로젝트 생성/비활성화, 공통 코드 관리 |
| 프로젝트 관리자/스크럼 마스터 | 프로젝트 정보 관리, 리소스 등록, 목표/태스크/이슈 조정, 미팅 진행 |
| 프로젝트 팀원 | 개인 진행업무, 오늘 목표, 블로커, 태스크 상태 업데이트 |
| 참조 사용자 | 읽기 전용 조회, 회의 참관, 현황 확인 |

### 4.2 권한 원칙
- 플랫폼 관리자는 전체 프로젝트와 공통 설정을 관리할 수 있어야 합니다.
- 프로젝트 관리자는 본인 담당 프로젝트의 보드와 리소스를 관리할 수 있어야 합니다.
- 팀원은 본인 업데이트와 할당 태스크를 수정할 수 있어야 합니다.
- 참조 사용자는 조회만 가능해야 합니다.

## 5. 플랫폼 정보구조

### 5.1 상위 메뉴 제안
1. 프로젝트 홈
2. 프로젝트 목록
3. 스크럼보드
4. 리소스 허브
5. 회의 로그
6. 관리자 설정

### 5.2 기본 사용 흐름
1. 프로젝트를 등록합니다.
2. 프로젝트 팀원과 프로젝트 리소스를 연결합니다.
3. 프로젝트별 활성 스프린트 또는 운영 주기를 생성합니다.
4. 해당 프로젝트의 스크럼보드에서 목표, 태스크, 이슈를 관리합니다.
5. 미팅 시에는 선택된 프로젝트 기준으로 보드를 진행합니다.

## 6. 프로젝트 등록 기능 설계

### 6.1 프로젝트 기본 정보
프로젝트 등록 화면에는 아래 항목이 필요합니다.

| 항목 | 설명 |
| --- | --- |
| 프로젝트명 | 표시용 공식 이름 |
| 프로젝트 코드 | 고유 식별자 |
| 설명 | 프로젝트 목적 및 범위 |
| 상태 | 준비 / 진행중 / 안정화 / 종료 |
| 프로젝트 관리자 | 오너 또는 PM |
| 스크럼 마스터 | 미팅 진행 책임자 |
| 운영 방식 | 주간 운영, 1주 스프린트, 2주 스프린트 등 |
| 시작일 / 종료예정일 | 일정 범위 |
| 기본 목표 템플릿 | 신규 스프린트 생성 시 적용 |
| 기본 태스크 상태셋 | `To Do`, `In Progress`, `Blocked`, `Done` 등 |

### 6.2 프로젝트 목록 화면 요구사항
- 프로젝트명, 상태, 담당자, 최근 미팅일, 열린 blocker 수를 표시합니다.
- 프로젝트 검색과 상태 필터를 지원합니다.
- 즐겨찾기 또는 우선 프로젝트 고정 기능을 제공합니다.
- 최근 접근 프로젝트를 빠르게 다시 열 수 있어야 합니다.

## 7. 프로젝트 리소스 등록 기능 설계

### 7.1 리소스 유형
프로젝트 리소스는 유형별로 등록 가능해야 합니다.

권장 유형:
- `Repository`
- `Requirements`
- `Design`
- `API Spec`
- `Issue Tracker`
- `Defect Tracker`
- `CI/CD`
- `Dev Environment`
- `Test Environment`
- `Production`
- `Monitoring`
- `Meeting Notes`
- `Shared Drive`

### 7.2 리소스 등록 필드
| 항목 | 설명 |
| --- | --- |
| 리소스명 | 예: `Frontend Repo`, `운영 테스트 URL` |
| 리소스 유형 | 위 유형 중 선택 |
| URL 또는 경로 | 링크 대상 |
| 설명 | 용도 설명 |
| 접근 권한 메모 | 계정 필요 여부, 접근 주의사항 |
| 담당자 | 관리 책임자 |
| 표시 순서 | 빠른 접근용 정렬 |
| 상태 | 활성 / 보관 / 점검중 |

### 7.3 리소스 활용 방식
- 프로젝트 보드 상단에 `리소스 바로가기` 영역을 둡니다.
- 미팅 중 자주 여는 리소스를 상단 고정합니다.
- 태스크나 이슈 카드에 관련 리소스를 연결할 수 있어야 합니다.

## 8. 프로젝트별 스크럼보드 구조

### 8.1 메인 화면 구성
선택된 프로젝트 기준으로 아래 6개 영역을 보여주는 구성이 적절합니다.

1. 프로젝트 컨텍스트 바
2. 상단 요약 영역
3. 팀 목표 및 핵심 결과 영역
4. 개인별 스탠드업 영역
5. 태스크 보드 및 일정 영역
6. 팀 이슈 및 의사결정 영역

### 8.2 프로젝트 컨텍스트 바
프로젝트별 보드라는 점을 명확히 보여주는 상단 고정 정보입니다.

표시 항목:
- 프로젝트명
- 현재 운영 주기명
- 프로젝트 상태
- 스크럼 마스터
- 주요 리소스 바로가기
- 최근 회의 일시

### 8.3 권장 레이아웃
```text
+--------------------------------------------------------------------------------------------------+
| 프로젝트명 | 주기명 | 상태 | 스크럼마스터 | Repo | 요구사항 | 이슈트래커 | 테스트URL | 회의로그 |
+--------------------------------------------------------------------------------------------------+
| 전체 완료율 | 진행 중 | 지연 위험 | Blocker | 오늘 마감 | 미입력 인원 | 최근 결정사항 |
+--------------------------------------------------------------------------------------------------+
| 프로젝트 목표 / 이번 주 핵심 결과 / 공지사항                                                      |
+---------------------------------------+----------------------------------------------------------+
| 개인별 스탠드업 카드                   | 태스크 보드                                             |
| - 어제 한 일                          | To Do | In Progress | Review | Blocked | Done       |
| - 오늘 할 일                          |                                                          |
| - 이슈/지원 요청                      |                                                          |
+---------------------------------------+----------------------------------------------------------+
| 일정 캘린더 / 오늘 일정 / 임박 일정 / 지연 태스크                                                 |
+--------------------------------------------------------------------------------------------------+
| 팀 이슈 / 리스크 / 외부 의존성 / 의사결정 필요 / 회의 액션 아이템                                |
+--------------------------------------------------------------------------------------------------+
```

## 9. 섹션별 상세 설계

### 9.1 상단 요약 영역
팀이 미팅 시작 직후 확인해야 하는 핵심 상태만 노출합니다.

| 항목 | 설명 |
| --- | --- |
| 운영 주기명 | 예: `2026-04 2주차 스프린트` |
| 기간 | 시작일, 종료일, 남은 일수 |
| 전체 완료율 | 완료 태스크 수 / 전체 태스크 수 |
| 진행 중 건수 | 현재 진행 상태 태스크 수 |
| 지연 위험 건수 | 마감일 임박 또는 초과 태스크 수 |
| Blocker 건수 | 개인/팀 이슈 중 해결 필요 상태 건수 |
| 미입력 인원 | 스탠드업 업데이트 미작성 인원 수 |
| 최근 결정사항 | 가장 최근 회의 결론 요약 |

### 9.2 팀 목표 및 핵심 결과 영역
프로젝트별 스크럼보드가 단순 할 일 목록이 되지 않도록 목표를 분리해서 보여줍니다.

권장 항목:
- 운영 주기 목표 1~3개
- 핵심 결과 3~5개
- 목표별 담당 오너
- 목표 달성률
- 관련 태스크 묶음

### 9.3 개인별 스탠드업 영역
각 팀원의 발언 흐름을 고정하기 위한 카드 템플릿입니다.

권장 카드 필드:
- 이름
- 담당 영역
- 어제 완료한 일
- 오늘 목표
- 현재 진행 중 태스크
- 마감 임박 태스크
- 블로커
- 지원 요청 대상
- 자신감 상태: 정상 / 주의 / 위험

운영 원칙:
- 미팅 전 사전 입력을 기본으로 합니다.
- 미입력 인원은 상단에 별도 표시합니다.
- 카드 클릭 시 상세 업무 목록과 최근 업데이트 이력을 보여줍니다.

### 9.4 태스크 보드 및 일정 영역
프로젝트별 태스크 상태와 일정을 함께 확인해야 합니다.

권장 상태 컬럼:
- `To Do`
- `In Progress`
- `Review`
- `Blocked`
- `Done`

권장 태스크 카드 정보:
- 태스크명
- 담당자
- 목표 연결
- 시작일
- 마감일
- 예상 공수
- 현재 상태
- 지연 여부
- 관련 리소스 링크
- 외부 시스템 연계 ID

일정 보조 영역:
- 오늘 시작 태스크
- 오늘 마감 태스크
- 3일 내 마감 태스크
- 지연 태스크
- 캘린더 또는 타임라인 보기 전환

### 9.5 팀 이슈 및 의사결정 영역
개인 블로커와 팀 차원 이슈를 분리 관리하는 영역입니다.

권장 구분:
- 팀 이슈
- 외부 의존성
- 의사결정 필요
- 리스크
- 회의 액션 아이템

권장 필드:
- 제목
- 유형
- 설명
- 영향 범위
- 오너
- 목표 해결일
- 상태: Open / Watching / Resolved / Closed
- 관련 태스크 또는 관련 리소스

## 10. 스크럼 미팅 운영 흐름

### 10.1 미팅 전
1. 미팅 대상 프로젝트를 선택합니다.
2. 프로젝트 리소스와 열린 blocker를 확인합니다.
3. 각 팀원은 개인 카드의 `어제 한 일`, `오늘 할 일`, `블로커`를 업데이트합니다.
4. 프로젝트 관리자는 목표와 일정 변동 사항을 확인합니다.

### 10.2 미팅 중
1. 프로젝트 컨텍스트 바에서 프로젝트 상태와 주요 리소스를 확인합니다.
2. 상단 요약으로 전체 진행상황을 공유합니다.
3. 팀 목표 영역에서 이번 주 우선순위를 확인합니다.
4. 개인별 스탠드업 카드 순서대로 진행합니다.
5. 태스크 보드에서 일정 충돌과 병목을 확인합니다.
6. 팀 이슈 영역에서 의사결정 및 후속 액션을 확정합니다.

### 10.3 미팅 후
1. 액션 아이템 오너와 완료 목표일을 지정합니다.
2. 새로 드러난 blocker는 팀 이슈에 등록합니다.
3. 태스크 일정과 상태를 즉시 수정합니다.
4. 회의 로그를 저장하고 다음 미팅 기준점을 남깁니다.

## 11. 데이터 모델 제안
공통 플랫폼 기준으로 프로젝트, 프로젝트 리소스, 스크럼 운영 데이터를 분리하는 구성이 적합합니다.

### 11.1 신규 테이블
| 테이블 | 목적 |
| --- | --- |
| `platform_users` | 공통 사용자 정보 |
| `projects` | 프로젝트 기본 정보 |
| `project_members` | 프로젝트별 참여자와 역할 |
| `project_resources` | 프로젝트 리소스 정보 |
| `scrum_boards` | 프로젝트별 보드 기본 정보 |
| `scrum_cycles` | 스프린트/주간 운영 기간 관리 |
| `scrum_goals` | 운영 주기 목표 및 핵심 결과 |
| `scrum_tasks` | 프로젝트 태스크 관리 |
| `scrum_task_updates` | 태스크 상태 변경 이력 |
| `scrum_member_updates` | 개인 스탠드업 입력 데이터 |

## 12. 화면 동작 제안

### 12.1 기본 진입
- 사용자는 최근 접근 프로젝트 목록 또는 프로젝트 목록 화면에서 프로젝트를 선택합니다.
- 프로젝트를 선택하면 해당 프로젝트의 활성 운영 주기와 보드를 기본 진입 화면으로 엽니다.

### 12.2 프로젝트 전환
- 상단 프로젝트 스위처로 다른 프로젝트로 빠르게 전환할 수 있어야 합니다.
- 프로젝트 전환 시 해당 프로젝트의 리소스, 목표, 태스크, 이슈가 함께 갱신되어야 합니다.

### 12.3 미팅 모드
- 불필요한 편집 UI를 줄이고 발표 순서 중심 레이아웃으로 전환합니다.
- 미입력자, blocker 보유자, 지연 태스크 보유자를 우선 강조합니다.
- 프로젝트 리소스 바로가기를 상단 고정합니다.

### 12.4 관리 모드
- 프로젝트 정보 수정
- 프로젝트 리소스 등록/수정
- 목표, 태스크, 이슈 편집
- 운영 주기 생성/종료
- 회의 로그 확인

## 13. 다른 시스템과의 연결 원칙
- 스크럼보드는 독립 플랫폼으로 설계하고, 다른 시스템과는 링크 또는 연계 ID로 연결합니다.
- 특정 결함관리 시스템, 이슈트래커, 저장소 구조에 직접 종속되지 않도록 합니다.
- 필요 시 프로젝트 리소스로 외부 시스템을 등록하고, 태스크나 이슈에서 참조만 연결합니다.
