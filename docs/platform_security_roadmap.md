# 플랫폼 보안 단계 계획

## 1. 문서 목적

- 본 문서는 스크럼보드 프로젝트의 사용자 보안 구조를 단계적으로 도입하는 계획을 정리한다.
- 현재 단계의 단순 사용자 관리와 향후 플랫폼 구조 확장 시점의 Supabase Auth + RLS 도입 범위를 구분한다.

## 2. 결정 사항

- 현재 단계에서는 `Supabase Auth + RLS`를 필수 전제로 두지 않는다.
- 현재 단계에서는 `defect_manage`와 유사한 앱 내부 사용자 관리 방식을 우선 검토한다.
- `Supabase Auth + RLS` 기반 사용자 보안은 스크럼보드가 플랫폼 구조로 확대될 때 2단계 과제로 추진한다.

## 3. 단계별 운영 방향

### 3.1 Phase 1. 단기 MVP / 내부 운영 단계

- 목적: 빠른 업무 적용과 단순한 사용자 관리
- 기준 구조:
  - 앱 내부 사용자 테이블 기반 로그인/권한 관리
  - 관리자, 작업자 역할 구분
  - 서비스 계층에서 등록, 수정, 삭제 권한 제어
- 적용 이유:
  - 초기 사용성을 단순하게 유지할 수 있다.
  - OTP, 매직링크, 외부 인증 흐름 없이 빠르게 운영할 수 있다.
  - 기능 검증과 운영 방식 확정에 집중할 수 있다.
- 한계:
  - 프로젝트 단위의 강한 데이터 격리는 부족하다.
  - 데이터베이스 레벨 접근 통제가 약하다.
  - 다중 프로젝트, 다중 조직 운영 시 보안 요구를 충분히 만족하지 못할 수 있다.

### 3.2 Phase 2. 플랫폼 확장 단계

- 목적: 프로젝트별 데이터 격리와 역할 기반 보안 강화
- 도입 범위:
  - Supabase Auth 도입
  - `auth.users`와 앱 사용자 매핑 구조 정비
  - 프로젝트 멤버십 테이블 추가
  - 프로젝트 단위 RLS 정책 적용
  - 관리자 범위와 프로젝트 관리자 범위 분리
  - 감사 로그와 접근 이력 강화
- 적용 조건:
  - 여러 프로젝트를 하나의 플랫폼에서 운영할 때
  - 프로젝트 간 정보 노출 방지가 필수 요구사항이 될 때
  - 외부 조직 또는 다수 팀이 함께 사용할 때

## 4. Phase 2에서 추가할 보안 항목

- 사용자 인증: Supabase Auth 기반 로그인
- 사용자 식별: `auth.users.id`와 앱 사용자 프로필 연결
- 프로젝트 격리: 프로젝트 멤버십 기준 조회 제한
- RLS 정책:
  - 본인 데이터 조회/수정
  - 프로젝트 멤버만 프로젝트 데이터 조회
  - 프로젝트 관리자만 특정 관리 기능 수행
  - 플랫폼 관리자만 전체 프로젝트 관리
- 감사 항목:
  - 생성자, 수정자, 삭제자
  - 권한 변경 이력
  - 프로젝트 멤버십 변경 이력

## 5. 현재 단계에서의 개발 원칙

- 기능 구조는 향후 Auth + RLS 확장을 고려해 분리된 서비스 구조로 유지한다.
- 사용자, 프로젝트, 미팅 데이터 모델은 추후 프로젝트 멤버십 확장이 가능하도록 설계한다.
- 현재 단계 문서와 구현은 단순 운영을 우선하되, 플랫폼 확장 시 재설계가 필요한 부분을 명시적으로 남긴다.

## 6. 후속 작업 목록

- 현재 단계 사용자 관리 구조를 `defect_manage` 방식에 맞춰 단순화
- 프로젝트 단위 사용자/권한 모델 초안 작성
- Phase 2용 Supabase Auth + RLS 설계 문서 별도 분리
- Phase 2 전환 시 필요한 데이터 마이그레이션 항목 정리
